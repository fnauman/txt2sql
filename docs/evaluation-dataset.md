# Evaluation Dataset And Scoring

This repository evaluates against deterministic public demo datasets instead of
live private data. This is the canonical reference for the failure taxonomy, the
value-aware comparison spec, the current baseline, and known coverage limits.

| Asset | Path |
|---|---|
| Core public benchmark | `datasets/core-public.json` (9 cases) |
| Paraphrase public benchmark | `datasets/paraphrase-public.json` (9 cases) |
| Edge-case public benchmark | `datasets/edge-cases-public.json` (17 cases) |
| Value-aware comparator | `compareResults` in `src/benchmark.js` |
| Dataset verifier | `scripts/verify-dataset.js` |
| Edge-dataset generator | `scripts/build-edge-dataset.mjs` |
| Demo seed | `scripts/seed-public-db.js` |

The edge-case suite is `edge-cases-public.json` = the 9 reused `core_public_*`
cases plus 8 dedicated `edge_public_*` cases, regenerated by
`npm run build-edge-dataset`.

These datasets are synthetic regression fixtures, not a substitute for early user
feedback. In the original deployment, real users surfaced bilingual terms,
shorthand product names, and compact temporal phrases that a hand-authored
benchmark would not contain until the system was put in front of them. Treat
production/user feedback as the source of new cases, then distill public-safe
versions into these datasets. **Do not** add generated traces, private database
dumps, or customer/vendor-specific examples here.

## Failure taxonomy

Each dedicated edge case carries a `failure_class` tag naming the specific
weakness it targets. The 8 classes in `datasets/edge-cases-public.json`:

| `failure_class` | What it traps |
|---|---|
| `metric_column_confusion` | Picking the wrong amount column (gross vs net, header vs line) for a metric. |
| `grain_confusion` | Aggregating at document-header grain when the question is about line items, or vice versa. |
| `wrong_join_path` | Choosing a plausible-but-wrong join path between two tables. |
| `campaign_join_path` | Taking the correct multi-hop campaign path (sales line → product → campaign) instead of a wrong shortcut. |
| `wrong_date_column` | Filtering on the wrong date (e.g. posting date vs document date). |
| `stale_snapshot_field` | Reading an obsolete denormalized/snapshot column instead of the live source. |
| `master_data_resolution` | Resolving a fuzzy entity term ("sparkling water") to the right product(s). |
| `aggregation_shape` | Producing the wrong result shape — e.g. a long grouped rowset when the question asks for values pivoted into separate columns. |

Every gold query is **execution-verified** against the public demo database, so a
failing benchmark run is a model regression, not dataset rot (use
`verify-dataset` below to keep that true after schema/data changes).

## Comparison spec (value-aware scoring)

The legacy comparator (`compareRows`) keys each row by its exact lowercased
column name, so a correct answer with a different aggregate alias
(`total_net_sales_amount` vs `total_net_amount`) or an extra projected column was
scored as a `result_mismatch`. That cosmetic brittleness dominated real failures.

`compareResults` (`src/benchmark.js`) instead compares on **values**: it matches
the gold's compared columns to the model's columns by value — any name, any
position, extra predicted columns ignored — and checks the row tuples agree
(the Spider-style execution-match idea, allowing extra predicted columns).

A case opts in via a `comparison` block. **All** public cases carry one, so
value-aware scoring is active across every committed case; a dataset/case
*without* a `comparison` block falls back to the legacy exact-row behavior so
external datasets and `compareRows` callers are unchanged.

```jsonc
comparison: {
  mode: 'scalar' | 'rowset' | 'ranked',  // default 'rowset'
  compare_columns: [..gold column names], // default: all gold columns
  value_columns:   [..gold column names], // ranked: the ranking metric(s); default: gold numeric columns
  order: 'desc' | 'asc',                  // ranked: default 'desc'
  decimals: number,                       // rounding precision, default 2 (matches gold ROUND(.., 2))
  tolerance: number                       // absolute numeric tolerance; default 0
}
```

- Numbers match by equality after rounding to `decimals`, or — when `tolerance`
  is set — by true absolute difference (`|gold − actual| <= tolerance`).
- **scalar / rowset**: an order-blind bijection of compared row tuples must exist.
- **ranked**: that bijection must exist **and** the model's primary value column
  must be monotonic in `order` (catches "didn't sort / sorted wrong" while
  tolerating tie reordering by label). Ranked mode assumes `NULL` metrics sort
  last.

Mode distribution across the committed cases:

| Dataset | scalar | rowset | ranked |
|---|---|---|---|
| `core-public` (9) | 3 | 1 | 5 |
| `paraphrase-public` (9) | 3 | 1 | 5 |
| `edge-cases-public` (17) | 5 | 3 | 9 |

## Current baseline

The committed datasets are intentionally small and cover a limited set of
intents — treat their numbers as **smoke signals, not a reliability claim**. No
fixed pass-rate is committed, on purpose: generation is non-deterministic, so a
single `accuracy: 1.0` is a lucky sample, not evidence. Measure reliability with
repeated runs and report the honest lower bound:

```bash
npm run benchmark -- --dataset core-public --repeat 10
```

In `--repeat` mode the report adds a `reliability` block with per-run min/mean/max
accuracy, the overall pass-rate, per-case pass rates, and a **Wilson 95% lower
bound** on the true pass-rate (`wilsonLowerBound`, z = 1.96, in
`scripts/evaluate.js`). The Wilson lower bound is the headline number: 6/6 on one
run has a lower bound near 0.6, not 1.0. Reliability mode is a measurement and
never fails the process on expected run-to-run variance. Grow `datasets/` with
execution-verified gold queries before making any reliability claim.

## Coverage limits

Known gaps, verified in code — do not read more into the suite than it tests:

- **Temporal**: normalization handles **month-name + year** only (no relative
  dates, quarters, or YTD).
- **Master data**: resolution is **product-only**; other entity types are not
  resolved before generation.
- **Empty results**: no committed case has `expected_row_count = 0`. The pipeline
  *handles* a genuine zero-row answer (a dedicated empty-result insight card and
  an `empty_result_set` signal check), but the suite does not *target*
  empty-result robustness.
- **Guardrails**: the unqualified-identifier check only flags mixed-case unknown
  identifiers.

## Running it

Run locally after bootstrapping and seeding MariaDB. Validate every gold query
against the public demo database **without spending any LLM calls** (run this
after schema/data changes to separate dataset rot from model regressions):

```bash
npm run verify-dataset                            # all datasets in datasets/
npm run verify-dataset -- --dataset core-public
npm run verify-dataset -- --dataset edge-cases-public
npm run evaluate-retrieval -- --dataset edge-cases-public
```
